Taming Revocable Spot VM Instances with Diffused

A Linux runtime shim that lets a program’s execution roam across cloud spot instances — handling revocation automatically while keeping the spot discount.

Zhao Zhang · Zeying Zhu · Micah Sherr · Benjamin E. Ujcich · Wenchao Zhou

Georgetown University  ·  University of Maryland, College Park

International Conference on Software Maintenance and Evolution (ICSME) 2026

Read the Paper Code & Docs

Abstract

Modern clouds offer spot virtual machine (VM) instances that provide cost savings but leave unsolved technical debt, forcing users to handle unpredictable revocations. We introduce Diffused, a Linux runtime shim that enables process diffusion — the ability to seamlessly roam the execution of programs across instances. Diffused decouples revocation handling, allowing the use of spot instances as cheaper, diskless, and replaceable computation nodes.

We evaluate Diffused and show that it provides similar (or even improved) performance compared to on-demand instances while maintaining up to a 52% spot discount.

The key idea: process diffusion

Instead of designing every workload to survive revocation, Diffused splits a running process into two parts and lets the cloud handle the rest.

computational part

Roams freely across spot VMs

The CPU and memory state of the program — the part that just computes — is migrated onto cheap, powerful spot instances and recalled when they are revoked.

environmental part

Stays on a stable on-demand VM

Storage, file descriptors, and other environment interactions remain on a low-cost, non-revocable on-demand instance and are accessed remotely.

How it works

A program starts on the on-demand VM with DHost, a lightweight shim that lives inside the program’s own process space — no kernel changes and minimal application changes. A global monitor, DMon, watches all instances and decides when to act:

  • Roam. DMon moves a program’s computational part onto an available spot instance, which then runs it diskless using the on-demand VM’s resources via RPC.
  • Recall. On a revocation notice, the program is quickly pulled back to the on-demand VM — within seconds — and re-roamed once a new spot instance is available.

DMon roams the computational part onto spot instances and recalls it before revocation, while the on-demand VM keeps the environment consistent.

We evaluate Diffused on AWS EC2 across a representative sampling of real workloads — including FFmpeg, Memcached, llama.cpp, lbzip2, and the PARSEC suite — and compare it against migration-based approaches such as CRIU and Firecracker microVMs. Roamed programs match or exceed on-demand performance for CPU-bound work while flexibly combining on-demand and spot instances to stay both performant and cost-efficient.
